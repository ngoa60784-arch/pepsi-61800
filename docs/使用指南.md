# BreachWeave（tch-agent）使用指南

> 面向操作员的**项目简介与上手说明**。工程细节见 [ARCHITECTURE.md](../ARCHITECTURE.md)，部署见 [deployment.md](deployment.md)，授权作战与范围见 [engagement-mode.md](engagement-mode.md)。

---

## 1. 项目简介

**BreachWeave**（包名 `tch-agent`）是一套面向**已授权渗透测试 / 红队演练**的多智能体协作平台。

它不是「单个模型 + 工具列表」，而是把渗透流程拆成多个职责清晰的角色，由中央编排层统一调度：

| 角色 | 职责 |
|------|------|
| **指挥官（Commander）** | 你与系统对话的入口：注册目标、下达任务、上传历史记录、查看调度结果 |
| **调度器（Planner）** | 后台周期性自动调度：分配/回收 Solver、更新作战计划 |
| **求解器（Solver）** | 在隔离的 Docker 容器里执行杀伤链；实际打靶命令走远程 **Kali**（MCP `kali-arsenal`） |
| **观察者（Observer）** | 每个 Solver 内旁路 LLM：维护**本地**策略板，并把高价值结论**升格**到目标级共享黑板 |
| **验证者（Verifier）** | 对「目标已达成」类结论做独立复核 |
| **编排中枢（ChallengeManager）** | 纯逻辑层：目标、共享状态、Solver 生命周期，**不直接跑 exploit** |

**技术栈**：运行时 [Bun](https://bun.sh)；Web UI + REST/SSE；Solver 基于 `pi-coding-agent` SDK；LLM 全部走**外部 API**（本机无需 GPU）。

> **警告**：系统会对真实目标发起扫描与利用尝试。仅用于**已取得书面或流程授权**的目标，并配置作战范围（见下文）。

## 2. 架构一览

```
操作员 ──► Web UI（指挥官 / 目标 / 运行时 / 配置）
              │
              ▼
         ChallengeManager（编排）
         ├── Commander（对话）
         ├── Planner（定时 tick）
         ├── Verifier（复核）
         └── RuntimeManager
                  │
                  ▼
         Docker Solver 容器（本地）
                  │
                  └── MCP kali-arsenal ──SSH──► 远程 Kali（nmap / ffuf / …）
```

- **控制面**：本仓库进程（Web + 调度 + 状态），资源占用相对小。
- **执行面**：每个 Solver 一个容器；重活（扫描、爆破）在**远程 Kali** 上跑，通过 MCP 调用。
- **内置 MCP 脚本**在仓库 `mcp/` 目录；容器内路径为 `/opt/tch-mcp/*.py`（自动挂载）。

## 3. 环境要求

| 组件 | 要求 |
|------|------|
| 操作系统 | 开发/部署机建议 **linux/amd64**（Solver 镜像为 amd64） |
| Bun | ≥ 1.x，用于 `bun install` / `bun run web` |
| Docker | 本机运行 Solver 容器（需能拉取/构建 `tch-agent:latest` 镜像） |
| Python3 | 宿主机探测 MCP 时需要；内置 MCP 依赖 `asyncssh`、`mcp[cli]`、`httpx`（`bun run install` 会尝试安装） |
| 远程 Kali | SSH 可达；配置在 **配置 → MCP → kali-arsenal** |
| LLM API | 在 **配置 → 提供商 / 模型** 中填写密钥与模型偏好 |

## 4. 快速开始

### 4.1 安装与启动

```bash
# 克隆仓库后
cd /path/to/hksong   # 或你的仓库目录名

bun run install      # 安装依赖 + 尝试安装 MCP 用 Python 包
bun run web          # 启动 Web，默认 http://127.0.0.1:3000
```

开发时可用热更新：

```bash
bun run dev          # 等同 bun --hot … web
```

首次启动会：

- 在 `~/.tch-agent/config/` 种子化内置 Prompt、MCP 条目（若不存在）
- 同步 runtime 资源（Dockerfile、solver 二进制）
- 首次可能编译 solver 二进制或校验 Docker 镜像（需数分钟）

### 4.2 最小配置清单

1. **配置 → 提供商 / API 密钥**：填入所用 LLM 服务商密钥。
2. **配置 → 模型**：为 Commander / Planner / Solver / Observer 选用模型（或在 **配置 → 调度器** 设统一默认模型）。
3. **配置 → MCP → kali-arsenal**：填写 `SSH_HOST`、`SSH_USER`、`SSH_PASS`（或 `SSH_ALIAS`），点击测试连通。
4. （推荐）设置作战范围：见 [§7 作战模式与范围](#7-作战模式与范围)。

### 4.3 第一次打靶（指挥官）

1. 打开 **指挥官** 页。
2. 输入自然语言，例如：`https://example.com 测一下这个目标`。
3. 指挥官会注册目标、启动实例、派出 Solver；可在 **目标** / **运行时** 查看进度。
4. 也可在 **目标** 页手动创建目标并 **启动 Solver**（选择 Prompt，如 `redteam-recon-exploit`、`kimi-security`）。

## 5. Web 界面说明

入口：`http://127.0.0.1:3000`（可用 `bun run web -- -l 0.0.0.0:3000` 改监听地址）。

### 5.1 主导航

| 页面 | 作用 |
|------|------|
| **指挥官** | 对话式指挥；支持附件上传（渗透记录等）；可撤回消息回滚状态 |
| **目标** | 目标列表与详情：实例、Memory/Ideas/Findings、攻击时间线、暂停/恢复测试、手动完成/撤销 |
| **运行时** | 所有 Solver 容器：状态、日志、消息流、Observer 策略板 |

### 5.2 配置（侧栏「配置」）

| 子页 | 作用 |
|------|------|
| **调度器** | 最大并发 Solver 数、Planner tick 间隔、每容器 CPU/内存、统一默认模型（保存时同步全部 Agent 含 Planner）、Planner 提示词 |
| **提供商 / 模型** | API 与模型注册、连通性测试 |
| **工具** | 注册自定义工具 |
| **MCP** | `kali-arsenal`、`vuln-intel` 等；SSH 测试、Kali 工具检查、一键装环境 |
| **技能 / 提示词** | 内置与自定义 Skill、Prompt 管理 |

### 5.3 典型工作流

**路径 A — 指挥官驱动（推荐新手）**

```
指挥官下达目标 → Planner 自动分配 1～N 个 Solver → 运行时观察工具调用与发现
→ 目标页查看 findings / 攻击流 → 人工确认是否收工
```

**路径 B — 目标页手动控制**

```
创建目标 → 填写 entrypoint → 启动指定 Prompt 的 Solver → 必要时再开第二个 Solver 并行
→ report_finding 写入本地成果 → 完成/撤销完成
```

**路径 C — 仅 Planner 无人值守**

```
配置调度器 tick → 创建目标并置为 running → Planner 按作战计划自动 launch/steer
```

### 5.4 多 Agent 协作（推荐理解方式）

系统不是「多个模型群聊」，而是 **ChallengeManager 统一黑板 + 周期指挥官**：

```
操作员 / Commander
       │
       ▼
  Planner（~30s 一轮）── launch / steer / battlePlan
       │
       ▼
 ChallengeManager（目标级共享态）
   memory / ideas / state assets / findings
       │                    ▲
       │ seed + 广播 steer   │ report_finding / record_asset
       ▼                    │ Observer 升格（bridge）
 Solver A/B/C…（Docker）────┘
   各带 Observer → 本地 .observer/board/
```

| 层级 | 存放位置 | 谁主要写入 | 谁读取 |
|------|----------|------------|--------|
| **目标级黑板** | `~/.tch-agent/challenge/{id}/` | Solver `report_finding`、`record_asset`；Observer 升格；UI/指挥官 | Planner 快照、新 Solver 启动任务、Verifier |
| **Solver 本地板** | `~/.tch-agent/solvers/.../.observer/board/` | 该 Solver 的 Observer | 本 Solver 的 `memory_list` / `idea_list` |

**三条协作链路（实战中最常见）**

1. **发现上报**：`report_finding` → 写入 findings → 高信号行升格到目标 memory → **steer 广播**同目标其他 Solver（避免重复踩已证实的死路）。
2. **Observer 升格**：Observer 在本地记 fact/failure/credential → 满足规则时经 host-bridge 写入目标级 memory/ideas → 再次广播给并行 Solver。
3. **Planner 指挥**：每轮读取目标 memory、findings、phase、各 Solver 的 **Current Focus** → `planner_launch_solver`（handoff 简报）或 `planner_steer_solver`（不重启容器改方向）。

**并发与阶段约束（默认）**

| 约束 | 说明 |
|------|------|
| 挑战实例槽 | 最多 **3** 个目标实例同时 running |
| Solver 槽 | **配置 → 调度器** 的 `maxSolvers`（默认 3） |
| handoff 长度 | Planner `solverHandoff` / steer `message` **≤ 2000 字符**；细节应写在共享 memory/findings |
| `KALI_PROVISIONER` | **仅**在目标进入 `foothold` / `breakthrough` 后才会出现在 Planner 可选 Prompt；`untouched` / `recon` 阶段隐藏且 launch 会拒绝（避免浪费 Solver 槽装工具） |
| 远程执行 | 扫打外网目标用 **`kali-arsenal` MCP**，不是 `KALI_PROVISIONER` Prompt |

**Engagement 统计**：目标页 / `stats.json` 的 `correct_submission_count` 在作战模式下按 **有效 finding** 计数（有 writeup 或 verifier 状态、且非 `rejected`），不再因 `correct:false` 恒为 0。

## 6. MCP 与远程 Kali

### 6.1 内置 MCP

| 名称 | 脚本 | 用途 |
|------|------|------|
| `kali-arsenal` | `ssh_mcp.py` | 在远程 Kali 上执行命令、后台任务、上传下载 |
| `vuln-intel` | `vuln_intel_mcp.py` | CVE / 漏洞情报查询（可选 API Key） |

配置文件：`~/.tch-agent/config/mcp.json`。  
默认参数路径为 **`/opt/tch-mcp/…`**；Runtime 会把仓库 `mcp/` 只读挂载到该目录。

宿主机上 **配置页探测 MCP** 时，会自动解析到仓库内真实脚本路径。

### 6.2 配置步骤

1. 打开 **配置 → MCP**，编辑 `kali-arsenal`。
2. 填写 SSH 信息；使用 **SSH 测试** 确认 root（或具备 sudo 的）连通。
3. 可选：**工具检查**、**一键装环境**（在远程 Kali 上安装常用渗透工具）。
4. 修改 MCP 后，**重启或新建 Solver 容器** 使连接生效。

### 6.3 Solver 如何使用 Kali

- Solver **不在目标机上跑**；它在本地容器里决策，通过 `mcp_kali_arsenal_ssh_execute` / `ssh_exec_bg` 等在 **你的 Kali** 上执行。
- 长任务（全端口 nmap、目录爆破）应使用 **后台任务** + `ssh_job_poll`，避免阻塞上下文。
- 禁止用容器内 `bash` 直接扫外网目标（无工具、无路由）；本地 `bash` 仅用于整理笔记再 `ssh_upload`。

## 7. 作战模式与范围

系统默认处于 **Engagement（授权作战）模式**：不连接外部 CTF 判题平台；成果写在本地。

### 7.1 范围文件（强烈建议）

```bash
export TCH_ENGAGEMENT_SCOPE=/绝对路径/engagement-scope.json
```

格式见 [engagement-scope.example.json](engagement-scope.example.json)：`allowed_targets`、`out_of_scope`、`rules_of_engagement` 等。

> 当前范围为**软约束**（写入 Solver 任务上下文），尚未在命令层硬拦截；需配合人工审计。

### 7.2 成果、收工与验证

| 动作 | 行为 |
|------|------|
| **`report_finding`** | 写入本地 submission（finding）；提取高信号事实到目标 **memory**；向同目标其他 Solver **steer 同步** |
| **`objective_achieved=true`** | 先过**证据门**（见下）；通过则 `pending` 并启动 **Verifier** 在容器内复现；仅 `verified` 才自动收工停 Solver |
| **手动完成** | 目标页「标记完成」→ `finishChallenge`（停实例 + 停 Solver） |
| **停止实例** | `stopChallenge`：**保留 entrypoint**，仅释放实例槽；Planner 之后可再起实例 |
| **删除目标** | 若仍有 **running** 的 Solver 会**拒绝删除**；需先停 Solver |
| **暂停测试** | 目标页暂停 → 停止该目标下 Solver 并阻止新 launch；恢复后继续 |

**「服务器权限 / root」类目标**（描述中含 root、shell、服务器权限等）：

- 仅凭 JWT、管理后台、API 密钥等**不足以**通过 `objective_achieved` 收工；需 writeup/proof 中出现 **shell 级证据**（如 `uid=0`、`/bin/bash`、`whoami`+root、webshell 可执行命令等）。
- Verifier 二次复核；`rejected` 时 steer 告知该 Solver 继续推进，目标**不会**被误标完成。

**共享态怎么用**

- **Memory**：已证实事实、证据、失败边界（死路）、凭据线索。
- **Ideas**：攻击假设（Observer 主写）；仅 `verified` / `failed` 会升格到目标级。
- **State assets**：结构化 host / service / credential / session（`record_asset`）。
- **Findings**：带 writeup 的战果列表；Planner 调度与统计都依赖此项。

---

## 7.3 Planner 自动调度要点

- 默认 **每 30s** 一轮（**配置 → 调度器** 可调）；Web 进程启动后即运行，改代码后需 **重启 `bun run web`** 生效。
- 输入：未解目标 + 各目标 phase（`untouched` → `recon` → `foothold` → `breakthrough`）+ findings 数 + 死路计数 + 各 Solver **Current Focus**。
- 输出：`planner-last-round.json`（摘要、工具动作、**battlePlan** 跨轮作战意图）。
- 典型动作：`planner_launch_solver`（带 handoff）、`planner_steer_solver`（并行时去重，例如两路都在打 Redis 时 steer 一路改打 API）、`planner_set_plan`。
- 日志：启动 Web 的终端可见 `========== planner round #N ==========`；launch/steer 失败会打印 `challenge:planner-tool` 错误。

## 8. 数据目录

所有状态在运行用户的 **`~/.tch-agent/`**：

```
~/.tch-agent/
├── config/           # API 密钥、模型、MCP、Prompt、host-settings
├── challenge/        # 目标、memory、ideas、findings、Planner 状态
├── solvers/          # 活跃 Solver 会话、workspace、.observer
├── archive_solvers/  # 已归档 Solver
└── runtime/          # Dockerfile、编译后的 solver 二进制
```

备份建议优先 **`config/`** 与 **`challenge/`**。

## 10. 部署与安全

- **公网暴露**：必须设置 `TCH_AUTH_TOKEN`；详见 [deployment.md](deployment.md) 与 systemd 单元 `deploy/tch-agent.service`。
- **架构**：控制面轻量；并发 Solver 数 × (2～3 vCPU, 3～4GB) 估算资源；镜像约 **10GB**，多容器共享同一镜像。
- **外网访问**：默认仅 `127.0.0.1`；可用 SSH 隧道或 nginx（SSE 需 `proxy_buffering off`）。
- **镜像**：建议在 CI/本机构建后 `docker push`，服务器只 `pull`，避免在云上构建。

## 11. 故障排查

| 现象 | 处理 |
|------|------|
| MCP `can't open file …/mcp/ssh_mcp.py` | 确认 `mcp.json` 为 `/opt/tch-mcp/*.py`；**新建/重启** Solver；确认从仓库源码启动 Web（挂载 `mcp/`） |
| `Failed to reconnect to kali-arsenal` | 检查 MCP SSH 凭据、网络、Kali 上 `python3` 与依赖 |
| Solver 起不来 | `docker images` 是否有 `tch-agent:latest`；**运行时** 页 startup 日志 |
| Planner 不派 Solver | **配置 → 调度器** `maxSolvers`、tick；目标 `running` 且未暂停测试；是否已满 3 实例 / 3 Solver |
| `planner_launch_solver` handoff 失败 | handoff 超过 **2000 字**；缩短为 3～6 条 bullet，长内容写入 memory/findings |
| `KALI_PROVISIONER is blocked during recon` | 预期行为；recon 阶段用 `redteam-recon-exploit` / `kimi-security` 等，打靶仍走 `kali-arsenal` |
| 停实例后 entrypoint 没了 | 应已修复：`stopChallenge` 保留 entrypoint；若仍出现请升级并重启 Web |
| 删目标报错「active Solver」 | 先在**运行时**停止该目标下所有 Solver，再删除 |
| `correct_submission_count` 一直为 0 | 作战模式需 finding 带 **writeup** 或 verifier 状态；升级后刷新 stats / 重启 Web |
| Planner 某轮 `stopReason: error` | 常见为 steer/工具超时；下一轮通常会恢复；可看终端 `challenge:planner` 日志 |
| 8080 连不上 | 多为目标未监听，非平台故障 |
| Kali 上 `tee /root/workspace/...` 失败 | 远程无该目录；可 `ssh_execute mkdir -p` 或改输出到 `/tmp` |
| 改代码后行为未变 | **重启** `bun run web`（Planner、handoff 上限、KALI 过滤等在进程内加载） |
