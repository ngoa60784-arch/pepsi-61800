## 1. 产品定位

**BreachWeave**（npm 包名 `tch-agent`）面向**已取得书面或流程授权的**渗透测试、红队演练、HVV 等场景。它不是「单模型 + 工具列表」，而是将渗透流程拆成职责清晰的多智能体，由非 LLM 的编排中枢统一调度。

| 特性 | 说明 |
|------|------|
| 控制面 | 本机 Bun 进程：Web UI、Planner、状态存储，资源占用相对小 |
| 执行面 | 每个 Solver 一个 **Docker 容器**（本地 amd64 镜像）；扫打命令走 **远程 Kali**（MCP `kali-arsenal`） |
| LLM | 全部走**外部 API**，本机无需 GPU |
| 作战模式 | 默认 **Engagement**：本地 findings、范围白名单，不依赖外部 CTF 判题平台 |

> **警告**：会对真实目标发起扫描与利用。仅用于授权目标，并配置作战范围（见配置手册 §12）。

## 2. 角色与职责

| 角色 | 类型 | 职责 |
|------|------|------|
| **Commander（指挥官）** | LLM | 操作员对话入口：注册目标、下达任务、查看调度 |
| **Planner（调度器）** | LLM | 周期性自动调度：分配/回收 Solver、维护 battlePlan |
| **Solver（求解器）** | LLM | 容器内执行杀伤链；外网命令经 `kali-arsenal` 在远程 Kali 执行 |
| **Observer（观察者）** | LLM | Solver 旁路：本地策略板，高价值结论升格到目标级黑板 |
| **Verifier（验证者）** | LLM | 对「目标已达成」做独立复核 |
| **ChallengeManager** | 纯逻辑 | 目标、共享 memory/ideas/findings、Solver 生命周期 |

### 2.1 协作架构

```
操作员 / Commander
       │
       ▼
  Planner（默认 ~30s 一轮）── launch / steer / battlePlan
       │
       ▼
 ChallengeManager（目标级共享态）
   memory / ideas / state assets / findings
       │                    ▲
       │ seed + steer       │ report_finding / record_asset
       ▼                    │ Observer 升格
 Solver A/B/C…（Docker）────┘
   各带 Observer → ~/.tch-agent/solvers/.../.observer/board/
```

**三条主链路**

1. **发现上报**：`report_finding` → findings → 升格 memory → **steer** 同目标其他 Solver。  
2. **Observer 升格**：本地 fact/credential → host-bridge → 目标级 memory/ideas → 再广播。  
3. **Planner 指挥**：读 memory、findings、phase、各 Solver **Current Focus** → `planner_launch_solver`（handoff ≤2000 字）或 `planner_steer_solver`。

**并发约束（默认）**

| 项 | 值 |
|----|-----|
| 同时 running 的目标实例 | ≤ 3 |
| 同时 Solver 数 | **配置 → 调度器** `maxSolvers`（默认 3） |
| `KALI_PROVISIONER` | 仅 `foothold` / `breakthrough` 阶段；`recon` 阶段隐藏且 launch 拒绝 |
| 远程打靶 | 必须用 **`kali-arsenal` MCP**，非容器内 bash 扫外网 |

## 3. 技术栈与仓库结构

| 层 | 路径 | 说明 |
|----|------|------|
| CLI | `apps/cli/` | `web` / `solver` / `solver rpc` |
| Web | `packages/ui-web/` | React + `Bun.serve` REST/SSE |
| 核心 | `packages/core/` | config、challenge、runtime、solver |
| MCP 适配 | `packages/libs/pi-mcp-adapter/` | `mcp.json` 加载 |
| MCP 脚本 | `mcp/` | `ssh_mcp.py`、`vuln_intel_mcp.py`；容器内 `/opt/tch-mcp/` |
| 部署 | `deploy/` | systemd、`build-and-push-image.sh` |

**运行时**：Bun（非 Node）。依赖方向：`UI → Runtime → Service → Config`，禁止反向。

```
apps/cli/              命令行入口
packages/core/         @tch/core
packages/ui-web/       Web + API
packages/ui-tui/       终端 UI（部分场景未默认启用）
mcp/                   内置 MCP Python
docs/                  项目文档、配置手册、范围示例
```

## 4. 环境与安装

| 组件 | 要求 |
|------|------|
| OS | 开发/部署建议 **linux/amd64**（Solver 镜像为 amd64） |
| Bun | ≥ 1.x |
| Docker | 本机运行 Solver 容器（镜像约 **10GB**，多容器共享） |
| Python3 | 宿主机探测 MCP；`bun run install` 安装 `asyncssh`、`mcp[cli]`、`httpx` |
| 远程 Kali | SSH 可达，在 **配置 → MCP → kali-arsenal** 填写 |

```bash
git clone <repo> && cd hksong
bun run install
bun run web                    # http://127.0.0.1:3000
bun run web -- -l 0.0.0.0:3000 # 监听所有网卡
bun run dev                    # 热更新（改 server.ts 仍需重启进程）
```

首次启动：种子化 `~/.tch-agent/config/`、同步 runtime 资源、可能编译 solver 二进制（数分钟）。

## 5. Web 界面

### 5.1 主导航

| 页面 | 作用 |
|------|------|
| **指挥官** | 对话指挥；附件上传；消息撤回 |
| **目标** | 目标详情、Memory/Ideas/Findings、攻击时间线、暂停/完成 |
| **运行时** | Solver 容器状态、日志、消息流、Observer 板 |
| **配置** | 见 [配置手册.md](配置手册.md) |

### 5.2 典型工作流

**A — 指挥官驱动**：下达目标 → Planner 派 Solver → 目标页看 findings → 人工收工。  

**B — 目标页手动**：创建目标 → 填 entrypoint → 选 Prompt 启动 Solver → `report_finding`。  

**C — Planner 无人值守**：配置 tick → 目标 running → Planner 自动 launch/steer。

### 5.3 内置 Solver Prompt（节选）

| 名称 | 用途 |
|------|------|
| `kimi-security` | 通用授权渗透（含 pentest/recon skills + kali-arsenal） |
| `CHALLENGE_PLANNER` | 调度器（系统用，非手动启动 Solver） |
| `OBJECTIVE_VERIFIER` | 收工验证子代理 |
| `KALI_PROVISIONER` | 远程 Kali 补装工具（仅 foothold/breakthrough） |
| `VULN_RESEARCHER` | 漏洞研究子代理 |

可在 **配置 → 提示词** 复制、修改或新建 Prompt（YAML frontmatter + Markdown 正文）。

## 6. MCP 与远程 Kali

| MCP | 脚本 | 用途 |
|-----|------|------|
| `kali-arsenal` | `ssh_mcp.py` | 远程执行、后台任务、上传下载 |
| `vuln-intel` | `vuln_intel_mcp.py` | CVE/漏洞情报（可选 NVD/GitHub token） |

- 配置：`~/.tch-agent/config/mcp.json`（UI **配置 → MCP** 编辑）。  
- 容器内路径 `/opt/tch-mcp/*.py`；宿主机探测时自动解析到仓库 `mcp/`。  
- 修改 MCP 或 SSH/FOFA 后需 **新建或重启 Solver** 容器。  
- FOFA：UI 填写后 **同步 FOFA 到 Kali** → 远程 `/root/.pentest-keys/keys.env`（详见配置手册 §6）。

**使用原则**：长扫描用 `ssh_exec_bg` + `ssh_job_poll`；禁止在容器 bash 直接扫外网目标。

## 7. 作战模式（Engagement）

默认开启，无需开关。

| 维度 | 行为 |
|------|------|
| 目标来源 | UI 或 Commander `create_target` |
| 范围 | `TCH_ENGAGEMENT_SCOPE` 指向 JSON（**软约束**，写入 Solver 上下文） |
| 成果 | `report_finding` → 本地 submissions |
| 收工 | Verifier `verified` 或操作员手动完成；`objective_achieved` 有证据门控 |

范围模板：[engagement-scope.example.json](engagement-scope.example.json)。

**收工门控（服务器权限类目标）**：需 shell 级证据（`uid=0`、`whoami`+root 等）；JWT/后台 alone 不足。

**统计**：`correct_submission_count` 按有效 finding（有 writeup 或 verifier 状态、非 rejected）计数。

## 9. REST API 索引

认证：设置 `TCH_AUTH_TOKEN` 后，除 `/api/auth/*` 外需 Cookie 或 `Authorization: Bearer`。

| 领域 | 路径（节选） |
|------|----------------|
| 认证 | `GET/POST /api/auth/status|login|logout` |
| 配置 | `/api/config/api-keys`、`providers`、`model-prefs`、`mcp`、`host-settings`… |
| Kali | `POST /api/config/mcp/kali-ssh-test`（含 `mode: sync-keys`）、`kali-tool-check`、`kali-provision` |
| 目标 | `GET/POST /api/challenges`、`/api/challenges/:id/...` |
| 运行时 | `/api/runtime/solvers`、`/api/runtime/stream`（SSE） |
| 指挥官 | `/api/commander/message`、`/api/commander/stream`（SSE） |

完整路由见 [ARCHITECTURE.md](../ARCHITECTURE.md) §10。

## 10. 部署与运维

### 10.1 资源估算

| 并发 Solver | 控制机建议 |
|-------------|------------|
| 1–2 | 8 vCPU / 16GB / 100GB |
| 3–5 | 16 vCPU / 32GB / 200GB |
| 6–10 | 32 vCPU / 64GB / 300GB |

远程 Kali 仅需 SSH 可达，不跑 solver 进程。

### 10.2 镜像分发

在 CI/本机构建并 push，服务器只 pull（勿在云上构建 10GB 镜像）：

```bash
deploy/build-and-push-image.sh registry.example.com/sec latest
docker pull … && docker tag … tch-agent:latest
```

或在 **配置 → 调度器** 填写完整 `runtime.image` 路径。

### 10.3 systemd

```bash
openssl rand -hex 32   # 作为 TCH_AUTH_TOKEN
sudo cp deploy/tch-agent.service /etc/systemd/system/
# 编辑 Environment=TCH_AUTH_TOKEN、ExecStart、User
sudo systemctl enable --now tch-agent
```

### 10.4 外网访问

- 默认 `127.0.0.1:3000`；SSH 隧道：`ssh -L 3000:127.0.0.1:3000 user@host`  
- nginx 反代 SSE 需 `proxy_buffering off`、`proxy_read_timeout 3600s`  
- **公网必设 `TCH_AUTH_TOKEN`**，否则任何人可调度渗透 Solver

### 10.5 环境变量（运维）

| 变量 | 说明 |
|------|------|
| `TCH_AUTH_TOKEN` | 非空启用 Web 登录 |
| `TCH_ENGAGEMENT_SCOPE` | 作战范围 JSON 绝对路径 |
| `TCH_AGENT_SKIP_PYTHON_DEPS` | 跳过 MCP Python 依赖安装 |
| `TCH_AGENT_SKIP_INSTALL_SCRIPT` | 跳过 postinstall |

更多见 [配置手册.md](配置手册.md) §11。

### 10.6 升级

```bash
cd /opt/tch-agent && git pull && bun install
sudo systemctl restart tch-agent
# 镜像更新时重新 pull + tag
```

## 12. 故障排查

| 现象 | 处理 |
|------|------|
| MCP 找不到 `ssh_mcp.py` | 确认 `mcp.json` 为 `/opt/tch-mcp/`；重启/新建 Solver |
| `kali-arsenal` 重连失败 | 检查 SSH、Kali 上 python3 与 MCP 依赖 |
| FOFA 同步 404 | 重启 `bun run web`；前端走 `kali-ssh-test` + `mode: sync-keys` |
| Planner 不派 Solver | `maxSolvers`、目标 running、未满 3 实例 |
| `KALI_PROVISIONER blocked` | recon 阶段预期；用 `kimi-security` 等 |
| 改代码行为未变 | **重启 Web**（Planner、路由等在进程内加载） |
